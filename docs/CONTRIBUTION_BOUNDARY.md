# Contribution Boundary

## Purpose

Define the public-safe contribution boundary for the cipher-topology diagnostic project.

## Current framing

This project evaluates whether persistent-homology features can serve as auxiliary diagnostics for structure, bias, or control-condition differences in generated byte streams.

The current registered evidence supports a diagnostic and reproducibility claim: the pipeline can detect deliberately structured weak-control streams under the configured embeddings, persistent-homology summaries, and SHA-256 seeded baseline comparisons.

## In scope

- Reproducible stream generation
- Null and control-condition design
- Topological feature extraction
- Internal randomness diagnostics
- Failure-safe external-test status reporting
- Effect-size and distance-to-baseline summaries
- Diagnostic reporting
- Evaluation infrastructure

## Current supported evidence boundary

- Six-condition 64-replicate internal TDA evidence is registered for AES-CTR conditions, SHA-256 seeded baseline, OS CSPRNG, LCG weak control, and xorshift32 weak control.
- Eight-condition 64-replicate weak-control sensitivity evidence is registered for the same design plus `periodic_byte_weak` and `biased_byte_weak`.
- The deliberately periodic and biased weak controls separate strongly from the SHA-256 seeded baseline in the reviewed internal TDA effect-size summaries.
- External randomness comparison is not claimed when Dieharder is unavailable; the status is recorded as unavailable rather than completed.

## Out of scope

- Cipher breaking
- Exploit development
- Claims of practical cryptanalytic attack
- Claims of AES, Ascon, DES, or TDEA vulnerability
- Security certification or validation of cipher strength
- Completed external-randomness comparison when Dieharder was not run
- Undocumented or non-reproducible benchmark claims

## Required before stronger claims

1. Freeze corpus and controls.
2. Document null model.
3. Add baseline notebooks or scripts.
4. Add tests for input generation and output schemas.
5. Produce reproducible result tables and figures.
6. Complete external randomness testing in an environment with Dieharder or a documented equivalent battery.
7. Separately register and validate any future Ascon, DES, or TDEA comparator implementation before mentioning empirical results for those ciphers.

## Public language

Use diagnostic, benchmark, evaluation, sensitivity, and reproducibility language. Avoid exploit, vulnerability, security-certification, or cryptanalytic-break language unless a future validated result warrants a separately reviewed claim.
