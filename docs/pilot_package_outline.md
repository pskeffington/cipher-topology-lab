# Pilot Package Outline

## Package name

Cipher Topology Diagnostic Platform — bounded technical pilot

## Purpose

This pilot packages `cipher-topology-lab` as a reproducible diagnostic workflow and evidence-control system. The pilot is designed for generated byte-stream review, weak-control sensitivity checks, workflow reproducibility, and claim-boundary reporting.

The pilot does not make security-certification, vulnerability, or cryptanalytic claims. It produces a bounded technical report that explains what was run, what artifacts were produced, what patterns were observed, and what claims remain out of scope.

## Intended users

- AI security or model-evaluation teams that need disciplined generated-data diagnostics.
- Research labs that need reproducible computational evidence workflows.
- Technical due-diligence reviewers evaluating experimental software claims.
- Foundations or sponsors funding technical research infrastructure.
- Graduate or applied research programs needing auditable evidence chains.

## Pilot deliverables

| Deliverable | Description |
|---|---|
| Diagnostic protocol | Scope, configs, conditions, baselines, and claim boundaries. |
| Deterministic demo run | Small reproducible run suitable for inspection. |
| Evidence register | Artifact inventory, validation status, backend status, and limitations. |
| Platform report | Reviewer-readable summary of claim, evidence, gaps, and next steps. |
| Technical appendix | Config and run metadata sufficient for reproduction. |
| Final recommendations | Suggested expansion, deferral, or next pilot scope. |

## Pilot phases

### Phase 1 — Intake and scope lock

Output:

- scoped pilot question;
- approved claim boundary;
- selected config or demo profile;
- expected artifact list.

Acceptance criteria:

- pilot question is diagnostic, not certification-oriented;
- claim boundary is documented before execution;
- success and failure criteria are named.

### Phase 2 — Reproducible run

Output:

- generated streams or selected generated-data inputs;
- embeddings;
- topological summaries;
- internal diagnostic summaries;
- validation and consistency outputs.

Acceptance criteria:

- all outputs trace back to a config;
- missing optional dependencies are reported clearly;
- failed stages are distinguished from unavailable tooling.

### Phase 3 — Evidence registration

Output:

- evidence register;
- artifact inventory;
- platform report;
- limitations section.

Acceptance criteria:

- every major claim maps to an artifact or explicit limitation;
- no prohibited claim appears in public or sponsor-facing text;
- claim-boundary validator passes.

### Phase 4 — Final report and next action

Output:

- final technical report;
- next experiment recommendation;
- productization or deferral decision.

Acceptance criteria:

- report is understandable to technical and non-specialist reviewers;
- recommendations are bounded and evidence-linked;
- no security-certification language is introduced.

## Out of scope

- Security certification.
- Vulnerability disclosure.
- Cipher-breaking claims.
- Production assurance.
- Proof of randomness.
- Unsupported comparator claims.

## Pilot success criteria

A pilot is successful if it produces a clear, reproducible, claim-safe evidence package. A pilot does not need to produce a positive separation result to be valuable. Negative, null, unavailable, or ambiguous results are acceptable if they are documented accurately.

## Pricing posture

Until buyer discovery is complete, pricing should remain internal planning language. Package value should be framed around reproducible technical review, evidence-register production, and sponsor-ready reporting rather than outcome claims.

## Next implementation dependency

The platform should complete the following before offering this as an external pilot:

1. `make claim-check` passes on public/reviewer-facing docs.
2. `make platform-report` produces stable report and JSON outputs.
3. `make platform-demo` runs a deterministic smoke workflow and report.
4. A final report template exists.
5. External randomness comparison is either completed or explicitly deferred.
