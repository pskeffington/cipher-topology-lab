# Architecture Separation: Research Core, Provenance Platform, GIS Extension

## Purpose

This note prevents conceptual drift as `cipher-topology-lab` expands from a cipher/TDA research repository into a platform asset with revenue-path provenance tools.

The repository now has three intentionally separated layers:

1. **Research core** — topological diagnostics of generated cryptographic byte streams;
2. **Provenance platform** — cryptographic source and citation evidence for AI/RAG workflows;
3. **GIS/topology extension** — civic and infrastructure layer provenance for high-impact AI-security pilots.

These layers are related, but they should not be blended into one unsupported claim.

## Layer 1: Research core

### Role

The research core evaluates whether topological data analysis can serve as an auxiliary diagnostic for generated byte-stream structure under controlled conditions.

### Main artifacts

- experiment configs;
- stream generation;
- embeddings;
- persistent-homology features;
- randomness diagnostics;
- effect-size summaries;
- evidence registers;
- manuscript artifacts.

### Claim boundary

The research core does not claim to break AES, certify randomness, prove security, or discover a production cryptographic vulnerability.

## Layer 2: Provenance platform

### Role

The provenance platform reuses the repository's evidence-register discipline for AI systems that ingest sensitive files, datasets, code/configs, PDFs, repositories, and knowledge-base exports.

### Main artifacts

- RAG provenance manifest;
- source and chunk SHA-256 fingerprints;
- RAG answer citation verifier;
- buyer-readable provenance reports;
- claim-boundary language;
- pilot report generator.

### Claim boundary

The provenance platform proves traceability against a manifest. It does not prove source truth, answer correctness, privacy compliance, regulatory compliance, cybersecurity sufficiency, or model safety.

## Layer 3: GIS/topology extension

### Role

The GIS/topology extension applies source-provenance and topology-aware reasoning to municipal, public-works, emergency-management, and infrastructure data layers.

GIS is included because spatial infrastructure systems are naturally topological: roads, parcels, utility networks, boundaries, containment, coverage, adjacency, and dependency paths all have graph or topology structure.

### Main artifacts

- GIS provenance profile;
- candidate GIS layer classification;
- missing-sector report;
- municipal/infrastructure pilot framing;
- remediation backlog.

### Claim boundary

The GIS extension does not certify map completeness, infrastructure safety, public safety readiness, operational readiness, or infrastructure security.

## Conceptual connection

The clean connection is:

```text
Topological diagnostics
→ evidence-register discipline
→ cryptographic provenance control
→ AI/RAG source verification
→ GIS/infrastructure source-trust profile
```

The project should not be described as one system that uses cryptanalysis to secure maps. It should be described as a research-rooted evidence platform with a commercial provenance branch.

## Reviewer-safe language

Use this framing:

> `cipher-topology-lab` began as a reproducible research workspace for topological diagnostics of generated cryptographic byte streams. It is now extended with a clearly separated provenance platform that applies the same evidence-control discipline to AI/RAG source verification and GIS/infrastructure source-readiness pilots.

Avoid this framing:

> The project uses cipher topology to prove GIS infrastructure security.

## Revenue-path rule

Revenue-path work should remain modular. It can reuse utilities, reporting patterns, and evidence discipline from the research core, but it should not alter the empirical claims of the cipher/TDA experiments.

## Repository organization rule

Use the following conceptual ownership:

| Layer | Code/docs location | Buyer/reviewer purpose |
|---|---|---|
| Research core | `configs/`, `scripts/00_*` through experiment pipeline, manuscript docs | academic/research credibility |
| Provenance platform | `rag_provenance.py`, `rag_verifier.py`, pilot report docs | paid AI source-trust pilot |
| GIS extension | `gis_profile.py`, GIS profile docs | municipal/infrastructure wedge |

## Next architecture action

Future README and platform docs should reference this separation before introducing GIS or revenue-path deliverables.
