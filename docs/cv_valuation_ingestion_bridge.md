# CV Valuation Ingestion Bridge

## Purpose

This bridge defines how `cipher-topology-lab` should expose platform-readiness evidence to the broader CV valuation system.

The valuation layer should not infer product value from repository size alone. It should ingest a structured platform object that captures evidence level, claim boundaries, demo status, packaging gaps, and next milestone.

## Source artifact

Primary generated object:

```text
results/platform/platform_object.json
```

Generated by:

```bash
make platform-report
```

Validated by:

```bash
make platform-artifact-check
make claim-check
make platform-validate
```

## Required fields for valuation ingestion

| Field | Type | Valuation use |
|---|---|---|
| `schema` | string | Confirms compatible object format. |
| `repo_full_name` | string | Maps object to repository identity. |
| `platform_name` | string | Human-readable platform name. |
| `market_segment` | string | Assigns market lane. |
| `evidence_level` | string | Supports readiness scoring. |
| `demo_status` | string | Supports demoability score. |
| `current_claim` | string | Captures supported claim. |
| `claim_boundaries` | list | Reduces overclaiming and risk. |
| `platform_lanes` | list | Shows reuse pathways. |
| `evidence_items` | list | Links valuation to evidence state. |
| `packaging_gaps` | list | Converts gaps into next actions. |
| `next_milestone` | string | Identifies conversion gate. |
| `recommended_next_commands` | list | Gives reproducible commands. |

## Suggested valuation mapping

| Platform object signal | Suggested score effect |
|---|---|
| `schema` present and valid | enables platform-object scoring |
| `evidence_level` includes registered evidence | increases evidence score |
| `demo_status` includes available report/demo | increases demoability score |
| `claim_boundaries` present | increases defensibility score |
| `platform_lanes` has more than one lane | increases reuse score |
| `packaging_gaps` present and bounded | improves next-action clarity, but reduces readiness until resolved |
| `next_milestone` present | improves conversion-path clarity |
| `platform-artifact-check` passes | increases validation confidence |
| `claim-check` passes | increases public-readiness confidence |

## Recommended CV valuation interpretation

For `cipher-topology-lab`, the CV valuation tool should treat the repository as:

- market segment: `security_ai_platform`;
- maturity: platform candidate;
- confidence: high if report/object validation and claim check pass;
- strongest value signal: reproducible diagnostic workflow plus evidence-register controls;
- primary remaining gap: deterministic demo and external-review package;
- productization lane: reviewer demo package, sponsored diagnostic pilot, evidence-register engine.

## Do not ingest as

- a security certification product;
- a vulnerability scanner;
- a cipher-breaking tool;
- a production assurance tool;
- a proof-of-randomness engine.

## Ingestion sequence

1. Run `make platform-report`.
2. Run `make platform-artifact-check`.
3. Run `make claim-check`.
4. Read `results/platform/platform_object.json`.
5. Map object fields into the CV market valuation scorecard.
6. Preserve `packaging_gaps` as conversion actions rather than failures.
7. Record validated status in the portfolio valuation snapshot.
