\section{Background}
\label{sec:background}

\subsection{Symmetric-cipher output and randomness diagnostics}
\label{subsec:symmetric-cipher-output}

AES is specified in FIPS 197 as AES-128, AES-192, and AES-256, each using a 128-bit block size and key lengths corresponding to the numerical suffix. In counter mode, a block cipher is used to generate a keystream-like sequence by encrypting counter blocks and combining the resulting output with plaintext. This makes AES-CTR a practical setting for studying output streams while preserving a clear separation between cipher design, mode behavior, plaintext pattern, and generated bytes.

Randomness testing is often used as a preliminary diagnostic for cryptographic outputs, random-number generators, and pseudorandom streams. Statistical tests can identify some regularities, biases, repetitions, or distributional anomalies. They cannot substitute for cryptanalysis, formal proofs, or security reductions. This distinction is central to the design of the present study. The statistical tests and topological summaries considered here are diagnostic instruments, not security proofs.

\subsection{Reproducible baselines and sensitivity conditions}
\label{subsec:reproducible-baselines}

The current design separates deterministic reproducibility from non-deterministic sensitivity analysis. A seeded SHA-256 expansion condition serves as the primary reproducibility baseline for baseline-distance calculations because it can be regenerated exactly from documented inputs. This makes it suitable for registered comparisons, effect-size tables, and evidence-register traceability.

Operating-system cryptographic random output is retained as a sensitivity condition because it represents high-quality platform randomness but is not intended to be exactly reproducible across independent executions. Its role is therefore different from the SHA-256 seeded baseline. It provides a useful reference condition, but not a fixed baseline for exact replication.

\subsection{Weak controls and deliberately structured streams}
\label{subsec:weak-controls-background}

Weak controls are necessary for evaluating whether a diagnostic has positive-control sensitivity. The initial design includes LCG and xorshift32 generator outputs as weak-generator controls. The weak-control sensitivity extension adds periodic and biased byte-stream controls to test whether the pipeline responds to deliberately introduced structure. These controls are not intended to model realistic cipher failures. Their purpose is to provide known structured alternatives against which the diagnostic pipeline can be evaluated.

\subsection{Lightweight and legacy cipher extensions}
\label{subsec:lightweight-legacy-background}

Ascon provides a planned lightweight-cryptography comparison point because it was selected by NIST for constrained devices and standardized for authenticated encryption, hash, and extendable-output functions. DES and TDEA are reserved for deprecated legacy-comparator experiments only. These extensions broaden the roadmap without changing the current AES-CTR, SHA-256 baseline, OS CSPRNG sensitivity, and weak-control evidence base. No empirical Ascon, DES, or TDEA result is claimed in the current thesis scope.

\subsection{Persistent homology}
\label{subsec:persistent-homology-background}

Persistent homology studies how topological features such as connected components and holes are born, persist, and die across a filtration. In the present study, byte streams are converted to point clouds or explicitly configured cubical-complex inputs, and persistent-homology features are computed from those representations. The resulting features are interpreted as empirical diagnostics of structure in the transformed output stream.

For H0, the summaries primarily describe connected-component behavior across scale. For H1, the summaries describe loop-like structure where finite intervals are present. Persistence entropy summarizes the distribution of finite persistence intervals and provides a scalar feature that can be compared across conditions. In this thesis, these summaries are not interpreted as direct cryptographic properties. They are features extracted from configured representations of byte streams.

\subsection{Evidence registration}
\label{subsec:evidence-registration-background}

A recurring risk in exploratory computational diagnostics is over-interpreting ad hoc outputs. The repository therefore treats evidence registration as part of the method. Each registered run records its configuration, artifact scale, backend status, validation status, external-test availability, registered figures, and logs. This creates a claim-control layer: manuscript statements must trace to registered artifacts rather than to informal inspection alone.
